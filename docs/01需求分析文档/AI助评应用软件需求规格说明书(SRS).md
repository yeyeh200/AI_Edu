## 1. 引言

### 1.1 目的

本文档旨在详细描述AI助评应用（AI助力教学评价应用）的软件需求规格，为系统设计、开发、测试和验收提供明确、完整的技术依据。通过规范化定义功能需求、非功能性需求和接口需求，确保项目各参与方对系统需求达成一致理解，指导后续的技术实现和项目管理工作。

本SRS文档将作为以下活动的重要参考：
- 系统架构设计和技术选型
- 软件开发和编码实现
- 系统测试和功能验证
- 用户验收和项目交付
- 系统维护和功能扩展

### 1.2 范围

#### 1.2.1 项目范围

AI助评应用是一个基于人工智能技术的教学评价系统，主要功能包括：

**功能范围：**
- 教学质量评估：对教师教学质量进行多维度智能评估
- 课堂质量分析：实时分析课堂教学过程和质量
- 课程质量评估：综合评估课程内容和教学效果
- 数据集成处理：整合职教云、教务系统、竞赛系统三大数据源
- 智能分析引擎：基于大模型API和规则引擎的深度分析
- 可视化展示：提供数据大屏、仪表盘和报表系统
- 权限管理：基于角色的访问控制系统

**数据范围：**
- 职教云平台数据：46个数据表，涵盖教学过程、课堂活动、学习记录
- 教务系统数据：25个数据表，包含学生信息、课程信息、成绩信息
- 竞赛系统数据：12个数据表，涵盖竞赛申报、成绩管理、材料管理
- 总计：83个数据表，覆盖教学全流程数据

**用户范围：**
- 教师用户：查看个人教学质量评价结果和改进建议
- 学生用户：参与教学评价和查看课程质量信息
- 管理人员：监控整体教学质量状况和制定管理决策
- 督导专家：进行教学质量评估和专业分析

#### 1.2.2 范围边界

**范围内包含：**
- 三大系统数据的抽取、转换、加载（ETL）功能
- 基于大模型API的智能文本分析功能
- 规则引擎的基础指标计算功能
- Web端、移动端H5、数据大屏三端展示
- 标准化API接口和数据服务

**范围外内容：**
- 学生个人隐私数据的详细分析
- 教师人事管理和薪酬管理功能
- 财务管理系统集成
- 其他非教学相关的系统功能
- 硬件设备的采购和维护

- **完整的**：包含所有必要信息

## 2. 总体描述

### 2.1 产品前景

#### 2.1.1 产品定位

AI助评应用是面向职业教育领域的智能教学评价系统，定位于：
- **智慧教学评价平台**：基于人工智能技术的现代化教学评价工具
- **数据驱动决策支持系统**：为教学管理提供科学的数据分析支撑
- **教学质量改进工具**：帮助教师识别问题并提供改进建议
- **教育数字化转型样板**：为职业教育评价改革提供示范案例

#### 2.1.2 产品特色

**智能化特色：**
- 集成先进的大模型API，实现深度语义分析
- 结合规则引擎，确保基础指标计算准确性
- 智能推荐和预警机制，主动发现问题

**数据驱动特色：**
- 整合83个数据表，覆盖教学全流程
- 实时数据更新，保证分析结果时效性
- 多维度数据分析，提供全面视角

**用户友好特色：**
- 多端适配，支持Web、移动端、数据大屏
- 个性化界面，满足不同用户需求
- 可视化展示，直观易懂

#### 2.1.3 发展前景

**短期目标（6个月内）：**
- 完成样板工程建设，验证技术方案
- 在试点院系成功应用，获得用户认可
- 建立完善的数据分析和评价体系

**中期目标（1-2年内）：**
- 在全校范围内推广应用
- 扩展功能模块，覆盖更多教学场景
- 与其他教育系统深度集成

**长期目标（3-5年内）：**
- 建设成区域性的智慧教育评价平台
- 为其他院校提供可复制的解决方案
- 推动职业教育评价标准化发展

### 2.2 产品功能

#### 2.2.1 核心功能模块

**1. 教学质量评估模块**
- 教师教学质量多维度评估
- 教学活跃度、学生参与度、教学效果等指标分析
- 同行对比和历史趋势分析
- 个性化改进建议生成

**2. 数据集成处理模块**
- 职教云、教务系统、竞赛系统数据抽取
- 数据清洗、转换和标准化处理
- 数据质量监控和异常处理
- 统一数据仓库管理

**3. 智能分析引擎模块**
- 大模型API集成（DeepSeek、文心一言）
- 规则引擎基础指标计算
- 文本情感分析和主题提取
- 智能推荐和预警算法

**4. 可视化展示模块**
- 数据大屏实时监控
- 个性化仪表盘定制
- 多维数据可视化图表
- 自动化报告生成

**5. 权限管理模块**
- 基于角色的访问控制（RBAC）
- 细粒度权限配置
- 数据访问范围控制
- 操作日志审计

#### 2.2.2 辅助功能模块

**1. 数据驾驶舱模块**
- 作业提交情况统计
- 学生成绩及预警监控
- 顶岗实习数据展示
- 第二课堂活动分析

**2. 系统管理模块**
- 用户管理
- 系统配置
- 日志管理
- 备份恢复

### 2.3 用户特征

#### 2.3.1 教师用户

**用户画像：**
- 年龄：25-60岁
- 技术水平：中等，具备基本计算机操作能力
- 使用频率：每周3-5次
- 使用场景：课后反思、学期总结、教学改进

**主要需求：**
- 查看个人教学质量评价结果
- 了解教学过程中的问题和不足
- 获得个性化的教学改进建议
- 与同行进行教学经验交流
- 记录和展示教学成果

**操作特点：**
- 关注评价结果的公正性和准确性
- 希望操作简单直观
- 需要详细的解释说明
- 重视隐私保护

#### 2.3.2 学生用户

**用户画像：**
- 年龄：18-25岁
- 技术水平：较高，熟悉数字化应用
- 使用频率：每月1-2次
- 使用场景：课程选择、学习反馈、成绩查询

**主要需求：**
- 参与教学质量评价
- 查看课程质量信息
- 获得个性化学习建议
- 反馈学习体验和意见
- 了解教师教学特色

**操作特点：**
- 偏好移动端操作
- 重视界面美观和用户体验
- 需要即时反馈和互动
- 关注个人数据隐私

#### 2.3.3 管理人员

**用户画像：**
- 角色：教务处长、院系主任、教学管理人员
- 年龄：35-55岁
- 技术水平：中等到高
- 使用频率：每日使用
- 使用场景：质量监控、决策制定、管理分析

**主要需求：**
- 查看整体教学质量状况
- 监控教学质量变化趋势
- 进行教学管理决策
- 制定教学改进措施
- 评估教学改革效果

**操作特点：**
- 需要宏观数据概览
- 重视数据准确性和及时性
- 需要深度分析功能
- 关注系统稳定性和安全性

#### 2.3.4 督导专家

**用户画像：**
- 角色：教学督导、质量评估专家
- 年龄：45-65岁
- 技术水平：中等
- 使用频率：每周2-3次
- 使用场景：教学评估、质量分析、标准制定

**主要需求：**
- 获取客观的教学评价数据
- 进行深入的教学质量分析
- 提供专业的改进建议
- 参与评价标准制定
- 监督评价过程公正性

**操作特点：**
- 需要专业的分析工具
- 重视数据背后的含义
- 需要灵活的分析维度
- 关注评价结果的权威性

### 2.4 约束

#### 2.4.1 技术约束

**架构约束：**
- 必须采用微服务架构，支持模块化部署
- 前端必须支持响应式设计，适配多端设备
- 后端必须采用RESTful API设计规范
- 数据库必须支持ACID特性，保证数据一致性

**技术栈约束：**
- 前端框架：Vue.js 3.x 或 React 18.x
- 后端语言：Python 3.8+
- 数据库：MySQL 8.0+ 或 PostgreSQL 13+
- 缓存：Redis 6.0+
- 容器化：Docker + Docker Compose

**集成约束：**
- 必须与学校统一身份认证系统集成
- 必须支持职教云平台数据接口
- 必须支持教务系统数据接口
- 必须支持竞赛系统数据接口

#### 2.4.2 时间约束

**项目周期：**
- 总体建设周期：8周
- 试点运行期：1周
- 系统优化期：1周
- 验收交付期：1周

**关键里程碑：**
- 第1周：项目启动和需求确认
- 第2周：数据准备和ETL开发
- 第3-4周：智能分析引擎开发
- 第5-6周：系统开发和集成
- 第7周：测试优化
- 第8周：试点运行

#### 2.4.3 成本约束

**总投资约束：**
- 项目总投资：40万元
- 硬件投资：≤10万元
- 软件投资：≤8万元
- 人力投资：≤15万元
- 其他费用：≤7万元

**运营成本约束：**
- 年度运营成本：≤27万元
- 大模型API调用费用：≤12万元/年
- 人员维护成本：≤10万元/年
- 其他运营成本：≤5万元/年

#### 2.4.4 性能约束

**响应时间约束：**
- 页面加载时间：≤3秒
- 简单查询响应：≤2秒
- 复杂查询响应：≤10秒
- 大模型API调用：≤15秒

**并发约束：**
- 同时在线用户：≥500人
- 峰值并发处理：≥1000 TPS
- 数据库连接池：≥50个连接
- API并发调用：≥100次/秒

**数据量约束：**
- 单表数据量：≥100万条记录
- 关联查询：≥10表关联
- 数据处理批量：≥10万条/批次
- 数据存储容量：≥1TB

#### 2.4.5 安全约束

**数据安全约束：**
- 敏感数据必须加密存储
- 数据传输必须使用HTTPS
- 必须实施数据脱敏处理
- 必须定期进行数据备份

**访问安全约束：**
- 必须实施统一身份认证
- 必须基于角色的访问控制
- 必须记录所有操作日志
- 必须实施会话安全控制

#### 2.4.6 法规约束

**数据隐私约束：**
- 必须遵守《个人信息保护法》
- 必须符合教育部数据安全规范
- 必须保护学生个人隐私
- 必须实施数据使用授权

**教育法规约束：**
- 必须符合职业教育质量标准
- 必须遵循教育教学规律
- 必须保证评价的公平公正
- 必须支持教学改革需要

#### 3.1.1 教学质量评估功能

**3.1.1.1 教师教学质量评估**

**需求描述：** 系统应能够对教师的教学质量进行多维度综合评估，提供客观、准确的评价结果和改进建议。

**输入数据要求：**
- 职教云教学活动数据：`t_zjy_course_face_teach_*`系列表
- 教务系统课程数据：`EDU_COURSE_*`系列表
- 学生评教数据：学生评价文本和评分
- 同行评议数据：专家评价意见

**评估维度要求：**
1. **教学活跃度评估（权重25%）**
   - 课堂签到率：计算公式为（实际签到学生数/应到学生数）×100%
   - 教学资源发布频率：发布的资源数量/教学周数
   - 作业布置和批改情况：24小时内批改率、批改质量评分
   - 互动讨论参与度：师生互动次数/学生人数

2. **学生参与度评估（权重30%）**
   - 学生课堂参与度：参与课堂互动的学生数/总学生数×100%
   - 作业完成率：按时提交作业的学生数/总学生数×100%
   - 在线学习时长：学生平均在线学习时间统计
   - 互动积极性：学生主动发起互动的次数统计

3. **教学效果评估（权重35%）**
   - 学生成绩提升：学生平均成绩的提升幅度计算
   - 学生满意度：学生对教学的满意评分统计
   - 知识掌握度：基于考试成绩的知识点掌握程度分析
   - 能力培养效果：能力评估的提升幅度分析

4. **教学创新评估（权重10%）**
   - 教学方法创新：新型教学活动使用情况分析
   - 资源建设创新：教学资源质量和多样性评估
   - 教学模式改革：教学模式变化和创新程度
   - 技术应用创新：信息技术在教学中的应用效果

**输出结果要求：**
- 综合评分：0-100分的量化评分，保留2位小数
- 各维度得分：四个维度的独立评分和权重应用
- 同行业对比：与同专业、同课程教师的对比分析
- 历史趋势分析：个人教学质量的历次变化趋势
- 改进建议：基于分析结果的3-5条具体改进建议

**业务规则要求：**
- 评价周期：按周、月、学期三个时间粒度进行评估
- 数据阈值：活跃度数据少于5次时不进行评估
- 权重调整：不同课程类型可调整评估权重
- 结果展示：教师只能查看自己的评价结果

**需求描述：** 系统应实现基于角色的访问控制（RBAC），确保不同用户只能访问授权的功能和数据。

**权限级别要求：**
1. **系统管理员权限**
   - 系统配置：系统参数和配置管理
   - 用户管理：用户账号和权限管理
   - 数据管理：数据的增删改查权限
   - 日志管理：系统日志的查看和管理

2. **管理人员权限**
   - 查看权限：所管辖范围内的数据查看
   - 管理权限：教学管理相关操作权限
   - 报表权限：管理报表的生成和查看
   - 决策权限：基于数据的决策支持权限

3. **教师权限**
   - 个人数据：个人教学数据的查看权限
   - 评价查看：个人评价结果的查看权限
   - 报表生成：个人报表的生成权限
   - 建议反馈：建议反馈和申诉权限

4. **学生权限**
   - 参与评价：教学评价的参与权限
   - 查看权限：公开信息的查看权限
   - 反馈权限：意见反馈的提交权限
   - 个人数据：个人数据的查看权限

**权限控制要求：**
- 功能控制：基于功能模块的权限控制
- 数据控制：基于数据范围的权限控制
- 操作控制：基于操作类型的权限控制
- 时间控制：基于时间范围的权限控制

**3.1.5.2 数据权限**

**需求描述：** 系统应实现细粒度的数据访问控制，保护敏感数据的安全和隐私。

**控制范围要求：**
1. **按用户级别控制**
   - 个人级别：仅能访问个人数据
   - 班级级别：能访问所带班级数据
   - 院系级别：能访问院系范围数据
   - 学校级别：能访问全校范围数据

2. **按部门范围控制**
   - 本部门：仅能访问本部门数据
   - 相关部门：能访问相关部门数据
   - 下级部门：能访问下级部门数据
   - 跨部门：特定情况下的跨部门权限

3. **按时间范围控制**
   - 实时数据：当前时间数据的访问权限
   - 历史数据：历史时期数据的访问权限
   - 预测数据：预测分析数据的访问权限
   - 归档数据：归档数据的访问权限

4. **按数据类型控制**
   - 基础数据：基础信息数据的访问控制
   - 统计数据：统计汇总数据的访问控制
   - 分析数据：分析结果数据的访问控制
   - 敏感数据：敏感数据的特殊访问控制

**权限验证要求：**
- 实时验证：每次访问都进行权限验证
- 会话验证：基于会话的权限验证
- 操作验证：关键操作的权限再验证
- 异常处理：权限异常的安全处理
